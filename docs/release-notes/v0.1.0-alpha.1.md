# StreamVolume Guard Hub v0.1.0-alpha.1

Status: alpha tester package, not a stable V1.

This note is the source text to use if a GitHub pre-release is created later.
Do not create a tag or GitHub release unless that is explicitly requested.

## What This Alpha Tests

- Windows desktop mixer control through `WindowsSessionVolume`.
- Browser extension observation and `BrowserGain` when the site/browser exposes a usable signal.
- Local bridge on `127.0.0.1:47841`.
- Honest source classification with `origin`, `controlSurface`, `status`, and `isControllable`.
- One-shot automatic volume correction to avoid continuous fader movement.
- Manual target profiles: `Calme` around 40%, `Standard` around 70%, `Fort` around 100%.
- Read-only Global Output Monitor with RMS/peak/state for the default Windows output device.
- Unknown active output detection through `global_output.unknown_active`.
- `Stream Safe`, guided test mode, and in-app `Guide OBS` actions for manual validation.
- Panic, exclusions, local logs, and copyable tester reports.
- Reproducible self-contained Windows tester package generated by `tools/package-tester.ps1`, with a SHA256 checksum.

## Important Limits

- No custom audio driver.
- No master-volume automation from the Global Output Monitor. It observes only.
- `global_output.unknown_active` is an alert, not an automatic fix. It means the Windows mix is active but known visible sources do not explain it.
- No account, cloud sync, telemetry, or automatic log upload.
- No OBS meter or scene reading in this alpha. OBS is a manual visual validation path.
- A browser can still appear as one Windows mixer source. If multiple tabs play at the same time and no usable `BrowserGain` source exists, the Windows fallback moves the whole browser.
- `BrowserGain` is valid only when the extension can measure useful signal and reaches a clear calibration state such as `locked`.
- On Chromium browsers, protected tabs start with `media-html` when a web player is accessible, then may upgrade generically to `tabCapture` if the HTML path stays silent or unavailable while the tab is audible.
- `needs-user-action`, `restricted`, and `unsupported` are expected states, not hidden failures: they mean click to protect the tab, page/browser capture is blocked, or the browser does not support this capture path.
- `ObserveOnly`, `Unknown`, `skipped`, or `no-signal` means the source is visible but not directly controlled by the extension.
- If tab capture is audible but Web Audio stays silent with the desktop connected, diagnostics expose `fallbackRecommended=true` and `fallbackReason=tab-capture-no-signal`; the expected control path is the desktop Windows-volume fallback.
- In that fallback state, the extension button can remain active even when direct `BrowserGain` is not possible. Active means "observed" or "trying the generic capture path"; it is routed to desktop fallback only when the desktop bridge is connected.
- If media HTML fallback reports detected media but `mediaProcessed=0`, that state is visible but not directly controllable; the desktop Windows-volume fallback remains the expected control surface only when the desktop is connected.
- If a web audio page exposes no controllable media element at all, standalone Chromium can now try the generic `tabCapture` path when the tab is audible; if no usable signal is captured, diagnostics stay honest with `ObserveOnly` / no-signal details.
- YouTube and TikTok may fall back to global browser volume when direct `BrowserGain` is unavailable.
- The Windows executable is not code-signed in this alpha. SmartScreen can show a warning until a signing certificate is added.

## Manual Test Order

Test one source at a time first:

1. YouTube in browser.
2. TikTok in browser.
3. Spotify Web or Deezer Web.
4. VLC or local player.
5. Discord.
6. Spotify desktop if available.
7. OBS with meters visible, manually observed.
8. Combined browser plus desktop-app scenarios.

For every source, copy the app report after the test and confirm:

- `origin` is visible.
- `controlSurface` is visible.
- `status` is understandable.
- `isControllable` matches the actual control surface.
- Non-controllable sources are not described as controllable.
- `Sortie globale` shows Silent/Safe/Risky/Unknown or a clear loopback error.
- The Global Output Monitor does not move Windows master volume and logs no raw audio.

## Known Follow-Up

- Real alpha logs still decide whether YouTube/TikTok/Spotify Web are mostly `BrowserGain`, `ObserveOnly`, or Windows fallback in the user's browser setup.
- A local diagnostic export would make support easier than copying reports manually.
- OBS integration remains research after the desktop/browser alpha proves the workflow.
