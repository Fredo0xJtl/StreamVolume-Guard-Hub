## Incoherences Trouvees

- Le repo n'avait pas de `AGENTS.md` racine alors que la methode projet et les regles anti-mauvaise-surprise doivent guider toute la base hybride.
- Le cahier des charges desktop disait encore que l'extension et le desktop etaient des produits separes, alors que le dossier courant est une base hybride.
- Certaines docs actives utilisaient encore un angle `Desktop V1` trop etroit pour un repo qui contient desktop, extension et protocole.
- La roadmap extension presentait l'audio des applications PC comme un sujet futur separe, alors que `apps/desktop` existe deja dans le repo hybride.
- Les anciens plans dans `docs/superpowers/plans/` contenaient des chemins historiques vers `D:\Codex\App StreamVolume Guard Hub` et pouvaient etre confondus avec une consigne actuelle.
- Le plan de suite A-G n'etait pas encore materialise dans un document source de verite.

## Decisions Prises

- `docs/hybrid-architecture.md` reste la vision architecture produit.
- `docs/product-next-plan.md` devient le plan de suite courant.
- `docs/desktop-v1-cahier-des-charges.md` devient le cahier de la couche desktop, pas la promesse complete du produit hybride.
- `apps/browser-extension/docs/future-implementation-roadmap.md` doit pointer vers l'integration hybride et non vers un desktop a etudier plus tard.
- `docs/superpowers/plans/` devient explicitement une archive de travail.
- Les limites de controle doivent rester visibles : `ObserveOnly` ou `Unknown` valent mieux qu'une fausse promesse.
